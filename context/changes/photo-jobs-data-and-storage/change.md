## Notes

Roadmap entry F-01. Establishes the data + storage foundation for the Cloud AI path: a private Supabase Storage bucket, a `jobs` table with per-user RLS, a 24h on-success source-retention contract, a service-role-backed signed-upload helper, and shared entity/DTO types. Not user-visible; unlocks S-03 (gated upload), S-04 (Realtime result), and S-05 (daily cap).

### Implementation notes

- **2026-05-28, Phase 1**: `npm run lint` produces ~1022 pre-existing `Delete ␍` (CRLF) Prettier errors on the Windows checkout, baseline of master. Verified by stashing the Phase 1 changes — error count identical with and without them. SQL migrations are outside ESLint's globs, so Phase 1 contributes zero new errors. Success criterion 1.2 satisfied under the adapted intent "no new lint errors from Phase 1"; baseline CRLF cleanup tracked as a separate concern (out of F-01 scope).
- **2026-05-28, Phase 1 (grants tightening)**: First migration draft used `grant select, insert on public.jobs to authenticated;` and relied on the absence of an explicit `grant ... to anon` to mean "anon has nothing". Schema verification (psql) found that Supabase's bootstrap applies blanket SELECT/INSERT/UPDATE/DELETE/REFERENCES/TRIGGER/TRUNCATE to `anon`, `authenticated`, **and** `service_role` on tables created in `public` by default — the explicit grant line is additive, not displacing. The anon sanity check still passed (RLS blocked the INSERT with code 42501), but the plan's intent was defense-in-depth (no anon grants at all). Migration now does `revoke all on public.jobs from anon, authenticated;` then `grant select, insert ... to authenticated;`. After the revoke, anon SELECT also fails at the grant layer (HTTP 401 `permission denied for table jobs`) before RLS evaluates — two independent gates instead of one. Worth capturing as a recurring rule for any future migration in this repo (see lessons backlog).
- **2026-05-28, Phase 4 (grants refinement on top of Phase 1)**: Phase 1's first revoke pass also revoked from `service_role`, with a misleading comment "service_role intentionally has no grants here; it relies on bypass-RLS". This broke every admin-client code path — Phase 4 tests failed with `permission denied for table jobs` on `createPhotoJob`'s insert. **Postgres GRANTS and RLS are orthogonal gates**: `service_role` has the `BYPASSRLS` attribute (so RLS doesn't apply), but it still needs table-level grants to read/write. The Phase 1 migration was amended in-place (still pre-deploy) to `revoke all from anon, authenticated;` — leaving `service_role`'s blanket grants intact. Refined lesson for the backlog: "Revoke blanket grants from anon and authenticated for defense in depth; leave service_role's grants intact — BYPASSRLS removes the RLS gate but not the GRANT gate."
- **2026-05-28, Phase 4 (createAdminClient module split)**: Plan/Phase 3 placed `createAdminClient` in `src/lib/supabase.ts` next to the existing SSR `createClient`. The Phase 4 test run failed at vitest module-load with `Cannot find package 'astro:env/server'` — the F1-round1 "factory takes env as a parameter" fix made `createAdminClient` env-free, but the file still imports `astro:env/server` for the existing `createClient`, and JS module loading is all-or-nothing. **Fix: split the file.** `createAdminClient` moved to `src/lib/supabase-admin.ts` (no `astro:env/server` import). `createClient` stays in `src/lib/supabase.ts` (unchanged behavior; same astro:env import). Tests now import the admin factory from the env-free module. The original plan was wrong about co-locating them — splitting is the right architecture; this is a minor plan deviation noted here for future planning passes (a service-role client should live in its own module so tests can reach it without dragging in Astro virtual modules).
